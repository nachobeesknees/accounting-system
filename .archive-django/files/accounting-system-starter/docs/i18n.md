# Internationalization (i18n) — English and Spanish

The system supports English and Spanish (Uruguay) as user-selectable interface languages. Spanish UI strings ship as part of the Uruguay localization module (v2). The i18n framework itself is wired in v1 so the addition of Spanish is purely additive.

## In scope for v1 (English only)

- Django i18n framework wired up (`USE_I18N`, `USE_L10N`, locale directories, `gettext` calls throughout)
- All user-facing strings wrapped in `gettext` — translation files are populated but only English is filled
- Language preference field on users — defaults to English; no second language to choose from yet
- Locale-aware number/date formatting (the framework handles this regardless of language count)

## Ships with Uruguay localization module (v2)

- Spanish (Uruguay) translations of all UI strings
- Uruguayan accounting terminology (see glossary below)
- Uruguay-specific number/date formatting (comma decimal, period thousand, dd/mm/yyyy)
- Language toggle appears in user settings (becomes meaningful with two languages available)

The reason for this split: i18n framework is cheap to wire up but only valuable when there are actual translations. Wiring it in v1 ensures no rework when v2 ships Spanish.

## Out of scope for v1

**Data localization (deferred to v2):**
- Account names stored in one language only (per entity, typically English)
- Entity names, vendor names, customer names — stored as entered, not translated
- Journal entry descriptions — stored as entered

This means: if an account is named "Cash — Operating" in English, it appears that way regardless of the user's UI language. Translating master data is a major scope add and isn't worth it for an internal system where the accounting team can read both languages.

If a specific entity needs Spanish-only master data (e.g., a Mexican entity where everyone works in Spanish), that entity's accounts can simply be named in Spanish — the per-entity CoA model allows this naturally.

## Implementation

### Django's translation framework

Django's built-in `gettext` is the standard approach:

```python
from django.utils.translation import gettext_lazy as _

class JournalEntry(models.Model):
    STATUS_CHOICES = [
        ('draft', _('Draft')),
        ('posted', _('Posted')),
        ('reversed', _('Reversed')),
    ]
```

Templates:
```html
{% load i18n %}
<h2>{% trans "Trial balance" %}</h2>
```

JavaScript translations via Django's JSCatalog view for client-side strings.

### Locale switching

- User preference: `language` field on the user model, default 'en'
- Middleware sets the active language per request based on user preference
- Settings dropdown allows the user to switch language at any time
- Persists across sessions

### Translation file workflow

- Source strings in English (the development language)
- `manage.py makemessages -l es` extracts strings to `locale/es/LC_MESSAGES/django.po`
- Translations done by Adam directly (you're presumably fluent enough; otherwise contract a translator for ~$500 for the v1 string set)
- `manage.py compilemessages` produces the compiled `.mo` files
- Translation review is a release-gate step

### Number and date formatting

- Use Django's `{% load l10n %}` and `localize` filter for numbers
- Use `Babel` library for more complex locale-aware formatting (e.g., currency display)
- `USE_L10N = True` in Django settings
- User's locale comes from their language preference

### Currency formatting

- Currency *codes* (USD, EUR, HKD, etc.) are universal — not translated
- Currency *display* (symbol placement, decimal separator) is locale-aware
- Storage: always Decimal with currency code. Display: formatted per user locale.

## Costs and complexity tradeoffs

**What this adds to the build:**

- ~10-15% additional UI work to wrap strings in translation calls (largely mechanical; AI tools handle this well)
- Translation file maintenance (one-time creation, ongoing maintenance as new strings are added)
- Locale-aware formatting (mostly free with Django/Babel)
- Bilingual report templates (real work — duplicate template logic for each report type)
- Testing in both languages (real work — every screen tested twice)

**Realistic added time:** 3-4 weeks across v0.1 - v0.6. Significant but not prohibitive.

**What we explicitly skip:**

- Auto-translation of user-entered data (vendor names, descriptions, etc.)
- Multi-language master data for accounts/entities
- Translation memory tools / translation management platforms (overkill for two-language scope)
- RTL (right-to-left) language support (only relevant if adding Arabic/Hebrew later)

## Future expansion

If a third language is ever added (e.g., Mandarin, Portuguese, French), the cost is incremental:

- New `locale/<code>/` directory
- Translation pass on existing strings
- Test pass
- No architectural changes needed

The architectural decision is "i18n from day one, two languages at v1." Adding language N+1 is a normal feature, not a rebuild.

## Working rules for Claude Code

1. **All user-facing strings go through `gettext`.** No bare strings in templates, views, or models that the user will see.
2. **Hardcoded English in code is a bug.** Even if it's a temporary placeholder, wrap it.
3. **Tests run in both languages.** A test that passes in English but fails in Spanish reveals i18n bugs.
4. **Translation files committed with the feature.** Don't ship code that references untranslated strings.
5. **Date and number formatting always uses Django's locale-aware filters or Babel.** Never format dates with `strftime` in display code.
6. **Currency display via a shared helper** (e.g., `format_money(amount, currency, locale)`). Never inline.
7. **When in doubt about translation, ask.** Some accounting terms are tricky in Spanish (e.g., "trial balance" → "balance de comprobación" or "balanza de comprobación" depending on regional convention). Adam decides.

## Uruguayan Spanish — locked conventions

Spanish locale is **Uruguay (es-UY)**. This drives terminology, formatting, and address conventions.

### Locale code

- Django language code: `'es-uy'` in `LANGUAGES` setting
- Locale directory: `locale/es_UY/LC_MESSAGES/`
- Babel locale: `es_UY`

### Number formatting

- **Decimal separator:** comma (`,`)
- **Thousand separator:** period (`.`)
- Example: `$ 1.234.567,89` (where English would be `$1,234,567.89`)
- Configure via Django's `USE_L10N = True` plus Babel for currency-specific formatting

### Date formatting

- **Format:** `dd/mm/yyyy` (e.g., `30/11/2026`)
- **Long form:** `30 de noviembre de 2026`
- **Short month:** `30 nov 2026`
- Weeks start Monday (per ISO and Uruguayan convention)

### Currency formatting

- Uruguayan peso symbol: `$` (with `$U` or `UYU` to disambiguate from USD where needed)
- Space between symbol and amount: `$ 1.234,56` (not `$1.234,56`)
- For USD shown to UY users: `US$ 1.234,56` or `USD 1.234,56`
- For other currencies: always use ISO code prefix to avoid ambiguity

### Accounting terminology (Uruguay-specific)

Uruguayan accounting Spanish has some terminology that differs from Spain or Mexico. Translation choices for v1:

| English | Uruguay Spanish |
|---|---|
| Trial balance | Balance de saldos *(also "balance de sumas y saldos" formally)* |
| Balance sheet | Estado de situación patrimonial *(Uruguay favors this over "balance general")* |
| Income statement / P&L | Estado de resultados |
| Cash flow statement | Estado de flujo de efectivo |
| Journal entry | Asiento contable *(or just "asiento")* |
| General ledger | Libro mayor |
| Chart of accounts | Plan de cuentas |
| Accounts payable | Cuentas por pagar / Proveedores |
| Accounts receivable | Cuentas por cobrar / Deudores |
| Period | Período (with accent) |
| Close (period) | Cierre |
| Post (an entry) | Contabilizar / Registrar |
| Reverse (an entry) | Reversar / Anular |
| Reconciliation | Conciliación |
| Bank reconciliation | Conciliación bancaria |
| Debit | Debe |
| Credit | Haber |
| Fixed assets | Activos fijos / Bienes de uso |
| Depreciation | Amortización *(note: Uruguay uses "amortización" for both depreciation of fixed assets AND amortization of intangibles, where English distinguishes them)* |
| Accrual | Devengamiento |
| Deferred revenue | Ingresos diferidos |
| Intercompany | Intercompañía |
| Consolidation | Consolidación |
| Audit | Auditoría |
| Auditor | Auditor (m.) / Auditora (f.) |

Maintain this glossary as new terms come up. When in doubt, follow Uruguay's *Normas Contables Adecuadas* terminology rather than Spain's *Plan General Contable* or Mexico's NIF.

### Address convention

- **Formal:** use *usted* form in UI messages, error messages, confirmations, emails
- **Imperative form for buttons/actions:** use the infinitive ("Guardar", "Cancelar", "Cerrar período") rather than imperative ("Guarde", "Cancele") — this is the modern UI convention even in formal contexts
- **No tuteo or voseo** in system messages (even though Uruguay uses voseo in spoken/informal Spanish, software UI conventionally uses neutral formal Spanish)

### Examples

- Confirmation: "¿Confirma que desea cerrar el período?" *(usted form, formal)*
- Button: "Cerrar período" *(infinitive)*
- Error: "El asiento no está balanceado. Verifique los importes."
- Toast: "Asiento contabilizado correctamente."
- Empty state: "No hay asientos en este período."

### Tax and reporting terminology (out of v1 scope but worth noting)

Uruguay has specific tax frameworks:
- DGI (Dirección General Impositiva) — tax authority
- IRAE (Impuesto a las Rentas de las Actividades Económicas) — corporate income tax
- IVA (Impuesto al Valor Agregado) — VAT
- BPS (Banco de Previsión Social) — social security

If any entity in the group is Uruguayan and has DGI reporting obligations, those reports may need to be in Uruguayan Spanish and use these specific tax terms. Out of scope for v1 (tax reporting is external), but documented here so it's not a surprise.

### Translation review

Adam reviews translations. If contracting a translator for the initial pass, the spec is:
- Uruguayan Spanish (not Argentine, not Spanish from Spain)
- Accounting domain expertise required
- Modern UI conventions (infinitive for actions, usted for messages)
- Glossary above as the starting point

Estimated translation cost for the v1 string set: $400-800 for a competent translator on Upwork or similar; closer to $1,500-2,500 for a domain-specialist firm. Adam doing it directly is also viable given fluency.
